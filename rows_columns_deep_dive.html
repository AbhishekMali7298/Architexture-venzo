What rows and columns actually mean — the precise logic

This is the most misunderstood part. Rows and Columns do NOT directly set how many tiles appear on screen. They set how many tiles fit in one seamless repeat unit — which then determines tile size.

Rows = 3 means the canvas height holds exactly 3 complete tiles + 3 joints. Tile height = (canvasH − 3×jointH) ÷ 3

The canvas size is FIXED (set by the px/download settings). Rows and Columns divide that fixed canvas into a grid. More rows = smaller tiles. Fewer rows = bigger tiles.

Common mistake: thinking "Rows = 3" means "show 3 rows of tiles." It does — but only inside the single repeat unit. When tiled as a background it repeats infinitely.

The "32 × 32 inch" readout in the sidebar is the real-world size of this single repeat unit. That's what gets stamped repeatedly across your surface in a 3D renderer.

The exact canvas math — how tile size is derived

Step by step calculation, nothing hidden:

Step 1 — Canvas dimensions are decided first (fixed):
canvasW = cols × tileW + cols × jointW
canvasH = rows × tileH + rows × jointH
But wait — this is circular. Tile size IS what we're solving for. So it's actually solved the other way:
Step 2 — Real-world tile dimensions (set by Width/Height inputs):
tileW_mm = state.tileWidth (e.g. 215mm for a standard brick)
tileH_mm = state.tileHeight (e.g. 65mm for a standard brick)
jointH_mm = state.jointHorizontal (e.g. 10mm)
jointV_mm = state.jointVertical (e.g. 10mm)
Step 3 — Canvas size in mm derived from rows/cols:
canvasW_mm = cols × (tileW_mm + jointV_mm)
canvasH_mm = rows × (tileH_mm + jointH_mm)
This is what the "32 × 32 inch" readout shows — it's derived, not set directly.
Step 4 — Convert to pixels (for the canvas element):
SCALE = canvasPixelWidth / canvasW_mm (e.g. 512px / 890mm = 0.575 px/mm)
tileW_px = tileW_mm × SCALE
tileH_px = tileH_mm × SCALE
jointH_px = jointH_mm × SCALE
Step 5 — Tile positions in the loop:
for row 0..rows-1, col 0..cols-1:
  x = col × (tileW_px + jointV_px)
  y = row × (tileH_px + jointH_px)
  ctx.drawImage(src, x, y, tileW_px, tileH_px)

How offset/bond affects the row loop

Running bond, 1/3 offset, Flemish — these all change how the x-position is calculated per row:

Stack bond (no offset):
x = col × cellW
Every row starts at col=0 with x=0. Perfect grid.
Running bond (½ offset):
rowOffset = (row % 2 === 1) ? cellW / 2 : 0
x = col × cellW + rowOffset
Odd rows shift right by half a tile. This creates the staggered brick look. The tile that "overflows" the right edge is split and drawn again at x = overflowX - canvasW.
1/3 offset (Roman bond):
rowOffset = (row % 3) × (cellW / 3)
x = col × cellW + rowOffset
Shifts by 0, 1/3, 2/3 on rows 0, 1, 2 then repeats. The "Stretchers" control in the sidebar sets this divisor.
Herringbone (special case):
Rows/cols don't map 1:1 to tiles — they set the REPEAT UNIT SIZE.
Herringbone tiles are 2:1 ratio. One "row" in the UI = one herringbone unit (pair of tiles at 90°). The actual pixel computation rotates tile pairs ±45° around their center. Rows/cols here scale up how many herringbone units fit in the repeat.

The seamless guarantee — why rows/cols must be integers

If rows = 3.5, the canvas height = 3.5 × cellH. The pattern repeats every canvasH pixels. But at the seam, the 0.5 partial tile at the bottom does not match the start of the first full tile at the top — the seam breaks. This is why the inputs only accept integers and snap on blur.

The seamless trick works because: canvasH = rows × cellH exactly. When the browser tiles this canvas with background-repeat: repeat, the bottom edge pixel row is directly adjacent to the top edge pixel row of the next repeat — and because the tile grid was computed to exactly fill canvasH, they match perfectly.
