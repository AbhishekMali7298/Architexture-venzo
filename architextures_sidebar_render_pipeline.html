How pattern change triggers a full re-render
Core pipeline▼
1
User changes any control (slider, dropdown, input)
Event fires: input / change / click on sidebar element
2
State object mutated immediately
state.pattern.rows = 4 — plain JS object, no framework
3
scheduleRender() called
Debounced ~20ms + requestAnimationFrame — prevents thrashing on slider drag
4
Pattern algorithm computes tile grid
Pure function → returns array of {x, y, w, h, rotation} for every tile
5
Canvas cleared → drawImage() loop
For each tile: ctx.save → translate → rotate → drawImage(materialImg) → restore
6
Pixel adjustment pass (ImageData)
getImageData → per-pixel brightness/hue/saturation → putImageData
7
Joint rendering pass
Draws gap lines OR fills joint material image into joint regions
8
Edge + surface overlay passes
Profile shadows, finish texture composited at reduced opacity
9
Canvas → CSS background + 3D preview update
canvas.toDataURL() or CSS paint API → body bg tiles it; Three.js texture.needsUpdate=true
Total time: ~5–20ms on modern CPU. No server involved. The same render() function runs server-side (node-canvas) for high-res downloads — identical logic, just bigger canvas.
Pattern section — every control explained
Pattern▼
Pattern picker (rotate icon)
Opens the "Choose Pattern" modal. Patterns are categorised: Brick Bond, Paving, Parquetry, Geometric, Organic, Random, Roofing. Each pattern is a pure JS function that receives {rows, cols, tileW, tileH, angle, stretchers} and returns an array of tile positions. Changing pattern swaps this function — the rest of the render pipeline is identical.
state.pattern.id = 'herringbone' → recompute grid → re-render
Rows
Number of tile rows in the output canvas. Directly controls how many times the drawImage() loop iterates vertically. Higher = more tiles = smaller tiles at fixed canvas size. Affects perceived scale in the real-world preview.
state.pattern.rows = N → grid recomputed
Columns
Number of tile columns. Works identically to Rows on the horizontal axis. Combined with Rows, these two values determine the total tile count = rows × cols (plus partial tiles at edges).
state.pattern.cols = N
Weaves
Visible for basket-weave / parquetry patterns. Controls how many tiles form each "bundle" before the direction alternates. A weave of 2 = pairs of 2 horizontal then 2 vertical tiles alternating. The algorithm uses this to group tiles before applying the rotation offset.
state.pattern.weaves = N
Angle
Rotates the entire tile grid. Implemented as ctx.rotate() applied to the whole canvas context before the tile loop, then rotated back. At 45° all tiles draw diagonally — this is how the diagonal herringbone variant works.
ctx.rotate(state.pattern.angle * Math.PI/180)
Stretchers
For brick bond patterns: controls how many stretcher courses appear between header courses. A "running bond" has infinite stretchers (all stretchers). A "Flemish bond" alternates stretcher+header per row. This value changes which row-type pattern function is called for odd vs even rows.
Size readout (32 × 32")
Live display of the total canvas tile area in real-world units. Calculated from tile width/height × rows/cols + joint sizes, then converted using the unit scale factor (mm or inches). Updates on every render — purely a derived display value, not a state input.
Material section — every control explained
Material▼
Material picker
Opens "Choose Material" panel. Categories: Stone, Brick, Wood, Terrazzo, Concrete, Metal, Tile, Fabric, Wallpaper, Carpet, Surfaces, Finishes, Landscaping, Insulation, Organic, Acoustic. Selecting a material fetches the full-res JPG from CDN (if not cached), stores it as an HTMLImageElement in JS memory, then triggers a full re-render. All subsequent renders use the cached image — no re-fetch.
state.material.img = new Image(); img.src = cdnUrl; img.onload = render
Upload width (")
The real-world width of the uploaded/selected material image in inches. Critical for scaling: it tells the renderer how many pixels in the source image = 1 inch. If your granite image is 2000px wide and represents 24", the scale factor = 2000/24 = 83.3px per inch. This ensures tiles drawn at "215mm wide" use the correct slice of the source image.
Width / Height (")
Real-world dimensions of each individual tile. These are the main sizing controls. Internally converted to canvas pixels using the unit scale: pixelW = (tileWidthInches / uploadWidthInches) * sourceImg.width. The drawImage() call uses these pixel dimensions as the destination rectangle size.
Min Width / Min Height
For "Random" placement patterns, tiles can vary in size. Min Width/Height set the lower bound of the randomisation range. The algorithm picks a random value between Min and the main Width/Height for each tile individually using Math.random(). Without this, random patterns would be uniform-sized.
Light Intensity
Applies a directional lighting simulation. Implemented as a radial or linear gradient overlay at low opacity composited on top of the finished texture. Brighter areas simulate a light source hitting the surface. Uses ctx.globalCompositeOperation = 'overlay' or 'soft-light'.
Surface (None / vermiculated / roughGrooved / voids)
Overlays a pre-drawn surface texture pattern on top of the material tiles. These are small tileable canvas patterns (not images) drawn procedurally: vermiculated = irregular worm-like indentations, roughGrooved = parallel scratches, voids = random circular cutouts. Applied via ctx.drawImage() of a pre-rendered offscreen canvas at ~30–50% opacity.
Surface Scale
Scales the surface pattern canvas before compositing. Implemented by scaling the offscreen canvas dimensions. Smaller scale = more repetitions of the pattern = finer texture.
Adjustments (Levels, Hue, Tint)
Opens the Adjustments modal (Brightness, Contrast, Hue, Saturation, Invert). These are applied per-tile via getImageData/putImageData pixel manipulation. Each slider stores a value in state, and the pixel loop applies all adjustments in sequence per tile draw. Done after each drawImage() on that tile's region — not globally, which allows per-material adjustments when multiple materials are used.
Tint
Opens colour picker. Applies a colour overlay using ctx.globalCompositeOperation = 'multiply' + ctx.fillStyle = tintColour. Multiply blending darkens the image toward the tint colour while preserving texture detail. A white tint = no effect.
Edges
Opens "Edge settings" modal: Style, Scale, Width. Draws a border effect around each individual tile's perimeter. Implemented as a separate strokeRect() call per tile after drawImage(), with a semi-transparent stroke. Simulates the slightly darker or lighter edge where a tile meets a joint. Style controls whether it's a simple stroke, a shadow/highlight combo, or a chamfered bevel.
Profile
Opens "Surface Profile" modal with Profile type, Repetitions, Gradient options. Simulates 3D surface relief on the tile edge. Draws a gradient shadow on the inner border of each tile to simulate a raised or recessed edge. The gradient goes from dark (at the very edge) to transparent (toward tile centre). Repetitions = how many concentric rings of this gradient are drawn.
Finish
Opens "Surface Finish" modal. Composites a finish texture (polished, brushed, matte) over the material. Uses a pre-rendered noise or pattern canvas at low opacity with a specific blend mode. The Scale control resizes this finish pattern relative to tile size.
Tone variation
Per-tile brightness randomisation. For each tile drawn, a random brightness offset within ±toneVariation range is computed and applied via ImageData. This is what makes textures look non-repetitive — identical tile images are drawn at slightly different brightness levels. Higher value = more visible variation.
Randomise fill angle
Per-tile rotation randomisation. Instead of all tiles using the same rotation (from the pattern), each tile gets a random ±N° offset applied on top. The drawImage() call wraps each tile in ctx.save/rotate/drawImage/restore. Most useful for organic/stone patterns to break the repetitive look.
Add Another Material (Pro)
Adds a second material slot with its own: material image, placement mode (Random/Defined/Manual), frequency, placement rules, and all the same adjustments. The render loop now calls a placement algorithm per tile to decide which material to draw — the "Rules" define which rows/columns get material A vs B, and frequency sets the ratio.
Joints section — every control explained
Joints▼
Joint material picker
The joint (mortar/grout) area is the gap between tiles. This picker sets what fills that gap. Options: Solid Fill (pure colour), or a material image (concrete, stone, etc.). If an image material is chosen, it's drawn into the joint regions using a separate drawImage() pass with the joint width as the destination rect size.
Joint Tint
Colour overlay on the joint material. Works identically to the tile tint — multiply blend mode. For solid-fill joints this effectively IS the joint colour. For image joints it tints the material.
Joint Adjustments (icon)
Opens the "Joint Adjustments" modal — identical controls to material adjustments (Brightness, Contrast, Hue, Saturation, Invert) but applied only to the joint pixels. Lets you darken mortar independently of tiles.
Horizontal (") + lock icon
Horizontal joint gap width in real-world units. Converts to pixels via unit scale. The tile grid spacing formula is: cellWidth = tileWidth + horizontalJoint. All tile x-positions are offset by this gap. The lock icon links/unlinks horizontal and vertical joint sizes so they change together.
Vertical (")
Vertical joint gap height. Works identically to Horizontal on the Y axis. cellHeight = tileHeight + verticalJoint.
Shadow opacity
Draws a drop shadow inside the joint region to simulate depth between the tile surface and the mortar. Implemented as a semi-transparent gradient from the tile edge into the joint channel. Higher value = joint appears deeper/recessed.
Recess joints
Toggle that makes joints appear physically recessed (lower than tile surface). Adds a darker gradient along both sides of each joint line. Combines with Shadow opacity to create a convincing 3D mortar channel effect.
Concave joints
Changes joint cross-section from flat to curved (tooled/raked mortar profile). Draws a curved highlight/shadow pair along the joint — lighter at the top, darker at the bottom — simulating a concave mortar tool finish.
PBR maps — how bump / normal / roughness / displacement work
PBR (Pro)▼
Bump map
Generates a grayscale bump map from the finished texture canvas. Source image options: None (flat), Grayscale (uses luminance of the texture directly — brighter = higher), Find edges (runs a Sobel edge-detection filter on ImageData to highlight tile borders and surface details). The result is a second canvas exported alongside the main texture.
Normal map
Converts the bump/grayscale source into a RGB normal map. Each pixel's normal vector is encoded as (R=X, G=Y, B=Z). Computed from the bump map by calculating the gradient (slope) between adjacent pixels using a finite difference kernel. The "Edge depth" slider scales the Z component — higher = stronger/deeper normals.
Displacement map
Same generation logic as bump but intended for geometry displacement in 3D renderers. The "Geometry" slider controls how much black-to-white range is compressed. "Edge depth" amplifies the joint channels specifically, making mortar lines appear as physical recesses in 3D mesh displacement.
Roughness map
Generates a PBR roughness map. "Base roughness" slider sets a uniform base level (0=mirror, 1=fully diffuse). Source image overlays texture-driven variation on top of this base. The result is: roughnessPixel = clamp(baseRoughness + sourcePixelLuminance * opacity). Joints are typically kept at high roughness (near 1.0).
Metalness map
Same as roughness but for the metalness PBR channel. "Base metalness" = uniform level. Source image adds variation. Most architectural materials are at 0 metalness (dielectric) — this is mainly useful for metal tile/cladding materials.
Invert colors (per map)
Inverts the generated map: 255-pixel for each channel. Useful when your source image's dark areas should be "high" not "low" on the bump, or when a renderer expects inverted normals.
Placement rules logic (multi-material)
Placement▼
Placement: Random
Each tile independently rolls Math.random() and is assigned material A or B based on the Frequency value. Frequency = 0.3 means 30% chance of material B per tile. No spatial pattern — purely stochastic.
Placement: Defined (Rules)
Opens the "Placement Rules" modal. Rows and Columns each have Hit/Miss/Hit/Miss toggles plus Shift. This creates a checkerboard-style rule: "on row 1 use materialA, row 2 use materialB, row 3 materialA..." combined with column rules. Shift offsets the alternating pattern by N tiles. Internally a 2D lookup: getMaterial(row, col) = rowRule[row % rowPeriod] && colRule[col % colPeriod] ? matA : matB
Placement: Manual
User can click individual tiles in the preview to assign material A or B to that specific tile. The state stores a sparse map of {tileIndex: materialId} overrides. The render loop checks this override map before the random/defined logic.
Frequency
In Random mode: probability (0–1) of each tile getting the second material. In Defined mode: the density/period of the Hit pattern — higher frequency = more tiles use material B.
Settings panel — global controls
Settings▼
Units (mm / inches)
Sets the global unit scale factor. All dimension inputs (width, height, joint size) display in this unit. Internally everything is stored in a base unit (likely mm), and all display values are converted on render. Switching units reconverts all displayed values without changing the actual stored dimensions.
Tile in background
When ON: takes the rendered canvas and sets it as the CSS background-image of the page body using background-repeat: repeat. The browser handles infinite tiling for free. When OFF: shows only the single tile canvas with no repetition. This is purely a CSS toggle — no re-render needed.
Show image border
Toggles a visible blue dashed border around the single tile canvas. Implemented as an absolutely-positioned div overlay. Helps visualise where the seamless repeat boundary is.
What makes it seamlessly tileable — the key math
Seamless logic▼
A
Canvas size = exactly N complete tile+joint cycles
canvasW = cols × (tileW + jointH) — no partial tiles at right edge
B
Offset tiles (running bond) wrap at boundary
Half-tile at the right edge = same half-tile at left edge. Modulo math: drawX = (x + offset) % canvasW. The tile is split and drawn twice — at the right edge AND at x=0.
C
Source material image is already tileable
CDN images are pre-prepared as seamless textures. The drawImage() crops a region that tiles correctly.
D
Tone variation uses a seeded random per tile
Random seed = hash(row, col). Same tile position always gets same brightness offset → the repeat looks consistent at the seam.
This is why "Rows" and "Columns" must be integers — fractional tile counts break the seamless wrap. The size readout (32×32") is always the exact tile area with no remainder.
